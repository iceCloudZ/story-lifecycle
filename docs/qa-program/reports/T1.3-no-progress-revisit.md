## 补充 1:shadow_router / debug_packet 是「死代码」,原报告未点破

原报告 Findings 把这两处引用定性为"相关但非 gate 逻辑,不满足 T1.3 代码落点"就带过了。
这个定性**对,但漏了更要紧的一层**:

| 位置 | 性质 | 关键事实 |
|---|---|---|
| `shadow_router.py:31/99/253` | 枚举触发器 `RETRY_NO_PROGRESS` | 在 no_progress **发生时**生成影子决策 |
| `debug_packet.py:61` | `payload.get("no_progress")` 读字段 | `_has_loop_exhausted` 据此判定卡住 |

**问题**:这两处是 no_progress 信号的**消费方**,但信号的**生产方已删**(detect_no_progress,ISS-008)。
全 src grep 无人写 `no_progress=True`。→ **这是死代码分支,永远不会触发**。

复核命令(原报告只 grep 了 `evaluation/`,漏了,要全 src):
```bash
grep -rn "no_progress" packages/story-lifecycle/src/ | grep -v __pycache__
# 结果:3 处全是消费方,无生产方
```

**影响**:不仅 README 过期,代码里还有 2 处等不到信号的死分支。建议**单开一张清理卡**(低优先级,
不影响功能)——删除 shadow_router 的 `RETRY_NO_PROGRESS` 触发分支 + debug_packet 的 `no_progress` 读取。

## 补充 2:原报告的「选项 B 补实现」建议不推荐,应优先选项 A

原报告交接 NOTE 给了 owner 二选一:
- **A. 文档修正**:README 删 no_progress 行
- **B. 补实现**:gate 加 progress 检测 + 测试

**建议明确走 A,B 不推荐。** 理由:

1. **max_retries 计数兜底已被验证有效**。T1.1(commit `ec0d2ec4`)刚证明 `round_count > max_retries`
   强制 fail 成立——这正是防死循环的兜底,FC 重写后已够用。
2. **no_progress 是 LangGraph 时代复杂度**。它要算"连续 N 轮 finding 无变化",需要存历史 finding、
   做差异比对——FC 简化的方向正是删掉这类状态机(ISS-008 删 detect_no_progress 是有意为之)。
   补回去等于回退 FC 重写的简化决策。
3. **语义重叠**。max_retries 已经能"修不动就停",no_progress 只是"没进步也停"——在 AI 自驱场景
   下,后者收益小、实现成本高。

走 A 的具体动作:`packages/story-lifecycle/README.md:93` 删除 `no_progress` 那一行(或在「收敛条件」
段注明"FC 重写后由 max_retries 计数兜底,no_progress 检测已移除,见 ARCHITECTURE.md §死代码已清")。
动作做完 T1.3 即可转 `[已完成]`(文档已对齐,非代码 bug)。
