# 验收报告模板与填写规范

> 每完成一张任务卡(`tasks.md`),产出一份验收报告存到 `reports/`。
> 报告是**下一窗口的输入**——它必须让一个全新会话(零对话历史)读懂「这张卡做了什么、结果如何、下一步接什么」。

## 文件命名

```
reports/<卡号>-<短名>.md
```

例:`reports/T1.1-gate-hard-fail.md`、`reports/T2.5-pty-injection-regression.md`

## 报告模板(复制此块填写)

```markdown
# <卡号> · <任务标题>

> 任务卡:[tasks.md#<锚>](../tasks.md)
> 日期:YYYY-MM-DD · 窗口:#<号>
> 状态:[已完成] / [阻塞] / [部分完成]

## 做了什么(What)

<2-5 句话,说清这个窗口实际干了什么。别复制任务卡的目标——写"我做了什么"。>

## 代码改动(Changes)

| 文件 | 改动 | 说明 |
|---|---|---|
| `tests/test_xxx.py` | 新建 | <测了什么> |
| `src/.../yyy.py` | 修改 | <为什么改;无则删行> |

commit: `<hash>`(若多窗口接力,列全部相关 commit)

## 验收结果(Acceptance — 必须实际跑过)

**命令**:
​```bash
<实际跑的 pytest 命令>
​```

**输出**(贴关键部分, PASSED/FAILED 计数 + 任何 warning):
​```
<原始输出>
​```

**结论**:✅ 全绿通过 / ❌ 失败(N 条)

## 发现(可选,Findings)

<执行过程中的观察:架构问题、可疑代码、文档与实现不符、可改进点。每条单独列。若没有写"无"。>

- **[观察/问题/风险]** <描述> — <影响> — <建议后续>

## 交接 NOTE(下一窗口接什么)

<这一段是报告最重要的部分——上下文满/中断时,新窗口靠它接续。>

- **本卡是否完成**:完成 / 未完成(还差 X)
- **下一窗口接什么**:<具体动作,如"继续 T1.2,fixture 已在 ... 准备好">
- **踩过的坑**(别让下个窗口重蹈):<如"mock invoke_with_tools 时必须同时 mock 返回 tool_calls 和 content,否则 planner NPE">
- **保留的上下文**:<关键变量名/文件路径/函数签名,新窗口不用重新 grep>
```

---

## 填写规范

### 1. 诚实优先(P4 协议)

- **验收输出必须实际跑过**。不许编造 PASSED。
- 失败了就写 `❌ 失败`,状态标 `[阻塞]`,留现场。
- 「应该能过」「估计没问题」这类话**禁止**——要么跑,要么标未验证。

### 2. 改动可追溯

- 「代码改动」表列**所有**改动文件(含测试)。
- 若改了生产代码(`src/` 下),必须说明**为什么改**(任务卡多数是"只测不改",若破了就是风险信号)。

### 3. 交接 NOTE 面向零上下文的新窗口

写交接 NOTE 时想象:**新窗口的 AI 只读了 `qa-program/` 三文件 + 本报告,它能继续干吗?**

不好的写法:
> 继续做 T1.2。

好的写法:
> 继续做 T1.2(gate 三分支)。我已建好 `test_gate_branches.py` 骨架,advance/retry 两分支已绿,
> **差 fail 分支**:fixture `gate_ctx_at_max_retries` 在 conftest.py:42,但调 `run_verify_gate`
> 时报 `KeyError: 'stage'`——gate.py:230 需要 context 里有 `current_stage` 键,fixture 没设。
> 下一窗口在 fixture 加 `"current_stage": "verify"` 即可。

### 4. 报告存档即不变

- 报告写完提交后**不修改**(它是历史记录)。
- 若后续推翻了某报告的结论,新写一份报告(T?.?- revisit)引用旧报告,说明推翻了什么。
