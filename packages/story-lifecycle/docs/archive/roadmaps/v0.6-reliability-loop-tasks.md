# v0.6 Reliability Loop 任务清单

> 目标：冻结当前功能面，先把“可安装、可启动、可跑通、可卡住、可排查”做成稳定闭环。

## 背景

当前代码已经超过早期 roadmap 的描述，诊断、Policy、Copilot、SWE-bench、Quality Flywheel 等能力都已经出现。但用户真实使用时，最先遇到的问题不是“智能度不够”，而是：

- 安装后 `story` 命令是否一定可用。
- `setup` / `doctor` 是否能在未配置、已配置、配置损坏时给出正确反馈。
- 最小 story 是否能稳定跑过 done 协议。
- 卡住时系统是否能明确说出卡在哪里。
- 用户是否能把足够的日志和诊断材料打包给维护者。

v0.6 因此不做新能力扩张，只做可靠性收敛。

## 范围

### 做

- 修复 CLI、TUI、诊断、done 协议、workspace 基础校验中的真实 bug。
- 补齐覆盖安装、启动、doctor、diagnostics、卡住识别的回归测试。
- 梳理已经实现的设计，把未实现内容移入后续 roadmap。
- 保证诊断包默认脱敏，且对维护者有足够信息。

### 不做

- 不新增 StoryOS 大能力。
- 不推进 Project Intelligence 自动接管。
- 不推进 Meta-Planner、Stage Graph、Graph Patch。
- 不扩展 SWE-bench 数据飞轮。
- 不做新的 LLM 对话式 Copilot 行为。

## P0 任务

| ID | 任务 | 验收标准 |
|----|------|----------|
| RL-01 | 安装与入口 smoke | editable install / wheel install 后 `story --help`、`story --version`、`python -m story_lifecycle --help` 可用 |
| RL-02 | setup / doctor 回归 | 未配置、已配置、配置损坏、环境变量覆盖四类状态输出清晰，不误报用法错误 |
| RL-03 | diagnostics 真实可用 | `story diagnostics STORY_KEY --no-zip` 输出可读诊断；zip 包含 `summary.md` 和 `debug_packet.json` |
| RL-04 | stuck reason 规则测试 | done 缺失、done malformed、CLI exited without done、stage timeout、loop exhausted 都有确定性 code |
| RL-05 | terminal output 进入诊断包 | 能捕获 recent output 时写入 `terminal/recent_output.txt`；不能捕获时 manifest 写明原因 |
| RL-06 | Board 右侧诊断面板回归 | 常规宽屏显示诊断摘要；窄屏自动降级；异常 story 不导致 TUI 崩溃 |
| RL-07 | workspace 基础校验 | 非 git repo、无写权限、旧 `.story-done`、多 workspace root 等场景给出明确提示 |

## 执行顺序

1. 先跑冻结基线：`ruff check src tests`、`pytest -q`、核心 CLI help 命令。
2. 先修安装和 CLI 入口，再修诊断包。入口不稳时，不继续扩 UI。
3. `story diagnostics --no-zip` 先可读，再做 zip 包和 TUI 渲染。
4. 每修一个历史卡住问题，都补一个回归测试。
5. 每完成一个 P0 任务，更新本文件状态或在 changelog 中记录。

## 建议验证命令

```powershell
$env:PYTHONPATH='src'
ruff check src tests
pytest -q
python -m story_lifecycle --help
python -m story_lifecycle setup --help
python -m story_lifecycle doctor --help
python -m story_lifecycle diagnostics --help
story --help
story --version
```

## 完成定义

v0.6 完成时，维护者应该可以对用户说：

1. 你可以安装。
2. 你可以启动。
3. 你可以跑一个最小 story。
4. 如果卡住，系统能说出一个确定性原因。
5. 如果还排查不了，你可以一键打包日志给维护者。
